今年 4 月，我买了智谱的 Coding Plan Max 套餐。

每周一上午 11 点，额度会重置成 100%。然后在这一周使用 AI 的过程中，Token 用一点、少一点。

所以我经常要做一件事——打开智谱官网，扫码登录，翻到后台的用量页，看一眼这周还剩多少。

![图 1：智谱 Coding Plan 后台用量页面](https://gudong.s3.bitiful.net/img/c4cff8bc.png)

为什么要看？因为我得把它用完，又不能用得太狠。用不完是浪费，用太狠下一周就要挨饿。

每次都是这套动作：打开浏览器、扫码、点进后台、找到用量页、看一眼、关掉。

30 秒。

但每天都 30 秒，一周三分钟半，一个月十几分钟。重点是这件事没有产出——**它就是一个单纯的损耗**。

某一天我又在重复这套动作的时候，突然想到——Mac 顶部菜单栏右上角，不是能显示股价、能显示天气吗？那种小工具。

那为什么不能让它显示我剩下的智谱额度？

我抬头看一眼，比打开浏览器快太多了。

## 1. 我把 API 喂给了 Claude Code

想到这，我就打开了 Claude Code。

但我不是直接说“帮我写个 MenuBar 工具”。我不知道 Mac 上这种东西怎么实现，也不知道有没有现成方案。

我做的事很简单——**先把接口给它**。

我打开智谱官网后台的用量页，按下 F12，调出浏览器的开发者工具，切到 Network 面板，刷新一下页面。

![图 2：浏览器开发者工具查看用量接口](https://gudong.s3.bitiful.net/img/c6f78362.png)

页面上那个“5 小时额度 12%、周额度 8%”的数据，背后一定有一个 API 在请求。

果然，我看到了 `bigmodel.cn/api/monitor/usage/quota/limit` 这个接口，返回的是一段 JSON，里面就是页面上显示的所有数字。

![图 3：智谱用量接口返回的 JSON 数据](https://gudong.s3.bitiful.net/img/0b5dd644.png)

我把这段 JSON 和接口路径，连同我的需求一起丢给 Claude Code：

> 我每天要看智谱 Coding Plan 的剩余额度，这是后台用量页的 API 接口和返回结构。帮我写个命令行工具，能查到周额度、5 小时额度、套餐到期时间。

它就开始干活了。

第一个版本是一个 Node.js 脚本。在终端里敲 `node zhipu-usage.js`，就打印出一份完整的用量报告：5 小时额度、周额度、套餐到期时间、最近 7 天调用次数、token 用量、模型分布。

很完整。

**但这个版本有个问题——我还要打开终端。**

打开终端、敲命令、看输出、关掉。虽然比浏览器快，但还是一个“主动去看”的动作。

我当时想要的是——它就在那儿，我不用主动看。

## 2. Claude Code 反过来教了我一课

我跟 Claude Code 说了这个想法。

它的回答让我有点意外——它问我：“你听说过 SwiftBar 吗？”

我说没听说过。

它说，SwiftBar 是 Mac 上的一个开源菜单栏扩展工具，可以在顶部菜单栏显示任意数据，只要写一个插件脚本就行。它推荐我用这个方案。

我原本以为，做这种事要自己开发桌面 App——学 SwiftUI、写界面、处理网络请求、打包签名——是个大工程。

**结果它告诉我，开源工具已经帮我把 90% 的路修好了，我只需要写一个 10 行的 shell 脚本。**

这是整件事里最让我惊喜的一段。

如果不是 Claude Code 主动提，我可能根本不会做这件事——因为在我心里“做菜单栏工具”等于“写一个 Mac App”，成本太大。

它继续干：在原来的 Node 脚本上加了一个 `bar` 模式，专门输出 SwiftBar 能识别的格式。又写了一个 8 行的 shell 包装脚本，调那个 bar 模式。

我把这个 shell 脚本放到 SwiftBar 的插件目录下，文件名按它的约定命名成 `zhipu-bar.10m.sh`——末尾的 `10m` 表示每 10 分钟刷新一次。

挂上菜单栏。

## 3. 现在我的菜单栏长这样

![图 4：Mac 菜单栏显示智谱额度](https://gudong.s3.bitiful.net/img/52e72d92.png)

Mac 顶部菜单栏右上角，永远显示着这样一串：

```
XX% │ XX%
```

前面是周额度，后面是 5 小时额度。抬头看一眼，就知道这周还能不能敞开用。

点开下拉，还有：周额度重置时间、5 小时额度重置时间、套餐到期日期、一个刷新按钮、一个跳转到智谱用量页的链接。

![图 5：SwiftBar 下拉菜单中的额度详情](https://gudong.s3.bitiful.net/img/ff9484a9.png)

我不需要主动查了。**它一直在那儿。**

如果你也想用：去智谱后台生成一个 API Key，写在 `.env` 里，剩下的让 Claude Code 帮你做。

## 4. 做小工具这件事，值不值得

这件事让我想了很多。

以前我也有过类似的小痛点——某个重复的操作、某个不顺手的流程、某个每天都要做的琐碎动作。

但我没做工具。为什么？

因为做一个小工具的成本太高了。

按传统流程做 Mac 菜单栏工具：学 Swift、学 SwiftUI、查菜单栏 API、写界面、处理网络请求、打包签名。一周起步。

为了一个 30 秒的小痛点，花一周写工具？

不值。所以这种小摩擦，过去我都是忍着。

但 Vibe Coding 把这个权衡打破了。

**现在做一个小工具的成本，是和 Claude Code 说几句话的时间。**

而且 Claude Code 还会主动告诉你“你不用从零开发，开源工具已经做完了 90%”——这一句话，比我研究半天方案还值钱。

由此我得到一个判断：

**AI 编程真正改变的不是“能做什么”，而是“做什么值得”。**

以前做不出来的小工具，现在能做了。以前忍着的小摩擦，现在不用忍了。

那些“自己做觉得不值，不做又难受”的小事，今天全都值得做了。

我觉得这也是普通人用 AI 最容易被忽略的地方。

我们周围每天重复的事情太多了。

有些事情不一定适合完全交给 AI，但也可以先问一句：能不能让它帮我做掉其中一小步？

哪怕一开始只省一点点，也值得。

随着 AI 的发展，随着这些工具越来越完善，一定会有越来越多重复的小事，可以被 Agent 接过去。

很多人现在还停留在“问 AI 问题”的阶段。

豆包、DeepSeek 这类聊天式 AI，很适合回答问题。

但如果你想让 AI 真正帮你省时间，就要多用 Agent。

Agent 不只是回答，它能写脚本、改文件、查资料、操作工具。

像 Claude Code、Codex、Work Buddy、小龙虾这类工具，才适合拿来处理我们每天重复的小麻烦。

你不用一开始就让 AI 做一件大事。

先从每天重复的小事开始。

哪个网页你总要打开，哪个数据你总要查，哪个流程你总要手动点。

让 Agent 先帮你做一点。

一次做不好就再改。

这些小地方省下来的时间，最后会很可观。
