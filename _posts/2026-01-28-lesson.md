![咕咚｜2026-01-29](https://gudong.s3.bitiful.net/weimd/1769612881996_image.png)
昨天的文章很有意思。

我写了那篇接近四千字的《[国内也能用！Claude Code 完整上手指南](https://mp.weixin.qq.com/s/Mg5c89lvARevhi5X3egIuA)》，发布之后，阅读量其实并不高，到现在还不到两千。

但**转发量**涨得特别快，已经快破 200 了。这应该是我近期转发率最高的一篇文章。

这也印证了一个现象：**大家确实喜欢“干货”。**

看到这种操作类的教程，大家的第一反应往往是先收藏，或者转发给朋友。能不能马上用到先不说，但“存下来”总归是好的。

说实话，看到这么多人转发，我挺欣慰的。毕竟，写教程这事儿，真的太费体力了。

## 1. 写教程，是件苦差事

熟悉我的朋友知道，我平时更喜欢分享感悟。

对于写教程，我其实内心是抗拒的。因为你要把脑子里的隐性知识，拆解成一步步的显性步骤：
*   这一步截图是不是必需的？
*   这个逻辑小白能不能看懂？
*   还要去验证流程有没有变。

昨天那篇文章，我足足花了两个多小时，写完只想躺平。

那为什么还要写？

一方面是因为答应了一个朋友（甚至被老婆催着要兑现承诺），另一方面，我也确实想把 **Claude Code** 这个好东西分享出来。

但我今天更想聊的，不是教程本身，而是一个关于**“如何学习”**的思考。

## 2. 关键词 > 教程

我不知道那 200 多个转发的朋友里，有多少人真的照着教程一步步去做了。

其实这不重要。

我觉得对于读者来说，看我的文章，**最大的价值不在于那几千字的步骤，而在于你获得了一个“关键词”。**

比如 **Claude Code**，比如 **CodeParty**。

我一直非常认可一个观点：**在 AI 时代，知道“有什么”，比知道“怎么做”更重要。**

你通过我的文章，知道了一个叫 Claude Code 的工具，知道它能改变编程方式，这就够了。

至于具体怎么安装、怎么配置，我写的可能不是最好的。你完全拿着这个关键词去搜，去问 DeepSeek，去问 ChatGPT：
> “什么是 Claude Code？国内怎么用？”

AI 能给你讲得更通俗，甚至能给你出更详细的步骤。

我自己平时逛推特、逛 GitHub 也是这样。我很少照着别人的教程一步步做，我是在**扫描关键词**。一旦发现了某个有价值的新东西，我就抓住这个词，然后用我自己的方式（通常是问 AI）去搞定它。

**教程只是引路人，关键词才是那把钥匙。**

## 3. 为什么我还要写？

既然大家可以问 AI，那我为什么还要费劲写那 4000 字？

因为**验证**。

工具的体验是会“失真”的。刚上手觉得好，不代表真的好，我刚上手的时候很兴奋，很想分享给别人，但没分享，因为当时的情绪是有明显的偏离客观事实的，认识也有一些偏颇。

然后我用 Claude Code 一个多月，后面开发了 **[SlideNote](https://mp.weixin.qq.com/s/nivLWAkYWiMHRHMENko0UQ)** 浏览器插件，重写了 **[inBox 笔记](https://mp.weixin.qq.com/s/l-EZl5MsXh-Y4uTbPAy80Q)** 的 PC 版。我是踩过坑、真的用它产出了东西，才敢把这个"关键词"抛给你们。

我不希望大家浪费时间。

所以在文章开头我也会诚实地劝退：如果你是新手，Claude Code 上手成本高，不如先试试 **Trae**。

## 4. 写在最后

虽然写教程很累，数据可能也没那么漂亮，但看到有那么多人转发，说明它是有价值的。

以后如果有更好的工具，发现了更好的“关键词”，我大概率还是会一边喊着“太累了”，一边老老实实地写一篇详细的教程分享给大家。

毕竟，**主动去寻找、去分享，这本身就是一种乐趣。**

希望大家都能通过这些关键词，找到适合自己的新大陆。
