# 为什么你的"AI 优先"战略大概率是错的

> 原文：Peter Pang（@intuitiveml），CREAO 创始人/CTO
> 翻译：小诸葛 🧠

---

我们 99% 的生产代码由 AI 编写。

上周二，我们上午 10 点上线了一个新功能，中午完成 A/B 测试，下午 3 点因为数据不达标就下掉了。下午 5 点，一个更好的版本上线。三个月前，这样的一个循环需要六周。

我们不是靠给 IDE 加 Copilot 实现的。我们把整个工程流程拆了重建，围绕 AI 重新设计。我们改变了计划、构建、测试、部署的方式，改变了团队里每个人的角色。

CREAO 是一个 Agent 平台。25 名员工，10 名工程师。我们从 2025 年 11 月开始构建 Agent，两个月前我从头重构了整个产品架构和工程工作流。

OpenAI 在 2026 年 2 月提出了一个概念，恰好描述了我们一直在做的事。他们称之为 **驾驭工程（Harness Engineering）**：工程团队的首要工作不再是写代码，而是让 Agent 做有用的事。当出现问题时，修复方式永远不是"更努力"，而是——缺了什么能力？如何让这个能力对 Agent 来说清晰可见且可执行？

我们自己得出了这个结论。只是当时没有名字。

## AI 辅助 ≠ AI 优先

大多数公司把 AI 拴在现有流程上。工程师打开 Cursor，产品经理用 ChatGPT 写需求，QA 尝试用 AI 生成测试。工作流没变。效率提升 10%-20%。结构上什么都没变。

这是 **AI 辅助**。

**AI 优先** 意味着你围绕"AI 是主要构建者"这个假设，重新设计流程、架构和组织。你不再问"AI 怎么帮助工程师？"，而是问"如何重构一切，让 AI 做构建，工程师提供方向和判断？"

区别是乘法级的。

我看到团队声称自己是 AI 优先，但还在跑同样的冲刺周期、同样的 Jira 看板、同样的站会、同样的 QA 签收。他们把 AI 加进了循环，但没有重新设计循环。

一种常见的版本是人们说的"Vibe Coding"——打开 Cursor，提示到能用为止，提交，重复。这能产出原型。但生产系统需要稳定、可靠、安全。当 AI 写代码时，你需要一个系统来保证这些属性。**你构建系统，提示词是一次性的。**

## 三个瓶颈

去年，我观察团队的工作方式，发现了三个会要命的瓶颈。

### 产品管理瓶颈

我们的产品经理花几周时间调研、设计、定义功能。产品管理几十年来都是这样运作的。但 Agent 可以在两小时内实现一个功能。当构建时间从几个月坍缩到几小时，几周的规划周期就成了瓶颈。

先想几个月，再两小时建完，这不合理。

产品经理需要进化为"有产品意识的架构师"，以迭代的速度工作；或者退出构建循环。设计需要通过"快速原型→上线→测试→迭代"的循环来完成，而不是委员会评审的需求文档。

### QA 瓶颈

同样的动态。Agent 交付一个功能后，我们的 QA 团队花三天测试边界情况。构建时间：两小时。测试时间：三天。

我们用 AI 构建的测试平台替代了手动 QA，用来测试 AI 写的代码。验证必须和实现在同一速度。否则你只是在旧瓶颈下游十英尺处建了一个新瓶颈。

### 人力瓶颈

我们的竞争对手用 100 倍以上的人做类似的工作。我们有 25 人。我们没法靠招人追平。我们必须靠重新设计来追平。

三个系统必须让 AI 贯穿其中：产品设计、产品实现、产品测试。任何一个停留在手动，都会制约整条管线。

## 先修代码库

我必须先修复代码库。

我们的旧架构分散在多个独立系统中。一个改动可能要涉及三四个仓库。从人类工程师的角度看，这还能管。从 AI Agent 的角度看，完全不透明。Agent 看不到全貌，无法推理跨服务影响，无法在本地跑集成测试。

我把所有代码统一到一个 **Monorepo**。原因只有一个：**让 AI 能看到一切。**

这是驾驭工程的一个实践原则。你能把系统中更多内容拉入 Agent 可检查、可验证、可修改的形式，你获得的杠杆就越大。碎片化的代码库对 Agent 是不可见的。统一的代码库是可见的。

我花了一周设计新系统：规划阶段、实现阶段、测试阶段、集成测试阶段。又花了一周用 Agent 重新构建整个代码库。

CREAO 是一个 Agent 平台。我们用自己的 Agent 重建了运行 Agent 的平台。如果产品能构建自己，那它就是可用的。

## 技术栈

### 基础设施：AWS

运行在 AWS 上，使用自动扩缩容容器服务和断路器回滚。如果部署后指标恶化，系统自动回退。

CloudWatch 是中枢神经系统。跨所有服务的结构化日志、超过 25 个告警、自动化工作流每日查询的自定义指标。每一块基础设施都暴露结构化的、可查询的信号。**如果 AI 读不了日志，它就没法诊断问题。**

### CI/CD：GitHub Actions

每个代码变更通过一个六阶段流水线：

验证 CI → 构建部署 Dev → 测试 Dev → 部署 Prod → 测试 Prod → 发布

每个 PR 上的 CI 门禁强制执行类型检查、代码检查、单元和集成测试、Docker 构建、通过 Playwright 的端到端测试、环境一致性检查。每个阶段都不可跳过，没有手动覆盖。流水线是确定性的，所以 Agent 可以预测结果并推理失败原因。

### AI 代码审查：Claude

每个 PR 触发三个并行的 AI 审查通道，使用 Claude Opus 4.6：

- **通道 1：代码质量。** 逻辑错误、性能问题、可维护性。
- **通道 2：安全。** 漏洞扫描、认证边界检查、注入风险。
- **通道 3：依赖扫描。** 供应链风险、版本冲突、许可证问题。

这些是审查门禁，不是建议。它们与人工审查并行运行，在大规模下捕获人类遗漏的问题。当你每天部署八次，没有人工审查者能在每个 PR 上保持注意力。

工程师还可以在任何 GitHub issue 或 PR 中 @claude，获取实现方案、调试会话或代码分析。Agent 能看到整个 Monorepo。上下文在对话之间保持。

## 自愈反馈循环：核心

这是整个系统的核心。

每天早上 UTC 9:00，一个自动化健康工作流运行。Claude Sonnet 4.6 查询 CloudWatch，分析所有服务的错误模式，生成一份执行层健康摘要，通过 Microsoft Teams 发送给团队。没人需要主动去问。

一小时后，分诊引擎运行。它将 CloudWatch 和 Sentry 的生产错误聚类，在九个严重性维度上对每个聚类评分，并在 Linear 中自动生成调查工单。每个工单包含样本日志、受影响用户、受影响端点和建议的调查路径。

系统会去重。如果已有开放的 issue 覆盖同一错误模式，它会更新那个 issue。如果之前关闭的 issue 重新出现，它会检测回归并重新打开。

当工程师推送修复时，同样的流水线处理它。三个 Claude 审查通道评估 PR。CI 验证。六阶段部署流水线依次通过 Dev 和 Prod 并在各阶段测试。部署后，分诊引擎重新检查 CloudWatch。如果原始错误已解决，Linear 工单自动关闭。

每个工具处理一个阶段。没有工具试图做所有事。每日循环创建了一个自愈循环，错误被检测、分诊、修复和验证，只需最少的人工干预。

## 功能开关与支撑技术栈

Statsig 处理功能开关。每个功能在开关后面上线。发布模式：先对团队开启，然后逐步百分比放量，然后全量发布或下线。关闭开关瞬间关闭功能，不需要部署。如果某个功能导致指标恶化，我们在几小时内拉掉。不好的功能在上线当天就死掉。A/B 测试通过同一系统运行。

Graphite 管理 PR 分支：合并队列 rebase 到 main 上，重新运行 CI，只在全绿时合并。堆叠 PR 允许在高吞吐量下增量审查。

Sentry 报告跨所有服务的结构化异常，由分诊引擎与 CloudWatch 合并以获得跨工具上下文。Linear 是面向人的层：自动创建的工单带有严重性评分、样本日志和建议调查方向。去重防止噪音。后续验证自动关闭已解决的 issue。

## 新功能路径

1. 架构师将任务定义为结构化提示，包含代码库上下文、目标和约束。
2. Agent 分解任务，规划实现，编写代码，并生成自己的测试。
3. PR 开启。三个 Claude 审查通道评估它。人工审查者检查战略风险，而不是逐行正确性。
4. CI 验证：类型检查、代码检查、单元测试、集成测试、端到端测试。
5. Graphite 的合并队列 rebase，重新运行 CI，全绿则合并。
6. 六阶段部署流水线依次通过 Dev 和 Prod 并在各阶段测试。
7. 功能开关对团队开启。逐步百分比放量。指标监控。
8. 出现恶化可随时拉掉关闭开关。严重问题断路器自动回滚。

## Bug 修复路径

1. CloudWatch 和 Sentry 检测到错误。
2. Claude 分诊引擎评分严重性，创建 Linear issue，附带完整调查上下文。
3. 工程师调查。AI 已经完成了诊断。工程师验证并推送修复。
4. 同样的审查、CI、部署和监控流水线。
5. 分诊引擎重新验证。如果已解决，工单自动关闭。

两条路径使用同一流水线。一个系统。一个标准。

## 结果

14 天内，我们平均每天 3-8 次生产部署。在旧模式下，那两周连一次生产发布都不会有。

不好的功能在上线当天就被拉掉。新功能在构思的当天就上线。A/B 测试实时验证影响。

人们以为我们在用质量换速度。用户参与度上升了。支付转化率上升了。我们产出了比以前更好的结果，因为**反馈循环更紧了**。每天发布比每月发布学到更多。

## 两种工程师

### 架构师

一到两个人。他们设计标准操作流程，教 AI 如何工作。他们构建测试基础设施、集成系统、分诊系统。他们决定架构和系统边界。他们定义对 Agent 来说"好"是什么样的。

这个角色需要深度批判性思维。你批评 AI，而不是跟随它。当 Agent 提出一个方案时，架构师找到漏洞。它漏了什么失败模式？跨越了什么安全边界？积累了什么技术债？

我有物理学博士学位。博士教我最有用的东西是如何质疑假设、压力测试论证、寻找缺失的东西。**批评 AI 的能力将比生成代码的能力更有价值。**

这也是最难填补的角色。

### 操作者

其他所有人。工作本身重要，但结构不同了。

AI 给人类分配任务。分诊系统发现一个 bug，创建工单，呈现诊断，分配给合适的人。那个人调查、验证并批准修复。AI 生成 PR。人类审查是否存在风险。

这些任务是 bug 调查、UI 调整、CSS 改进、PR 审查、验证。它们需要技能和专注。它们不需要旧模式所要求的架构推理。

## 谁适应得最快

我注意到一个出乎意料的模式：**初级工程师比高级工程师适应得更快。**

初级工程师因为传统实践较少，感到被赋能了。他们有了放大自己影响力的工具。他们不需要改掉十年的习惯。

高级工程师因为传统实践很强，适应最难。AI 一小时能完成他们两个月的工作。在多年构建稀缺技能后，这是一件很难接受的事。

我不做评判。我在描述我的观察。在这个转型中，**适应力比积累的技能更重要。**

## 管理坍塌了

两个月前，我花 60% 的时间管理人。对齐优先级。开会。给反馈。辅导工程师。

今天：不到 10%。

传统的 CTO 模式说要赋能团队做架构工作、培训他们、授权。但如果系统只需要一两个架构师，我需要先自己做。我从管理转向了构建。大多数日子我从早上 9 点编码到凌晨 3 点。我设计系统的 SOP 和架构。我维护整个驾驭体系。

更有压力。但我享受构建，而不是对齐。

## 更少争论，更好的关系

我和联合创始人及工程师的关系比以前更好了。

转型之前，我和团队的大部分互动是对齐会议。讨论权衡、辩论优先级、对技术决策有分歧。这些对话在传统模式中是必要的，但也很消耗。

现在我还是和团队聊天。我们聊其他的事。非工作话题。随意闲聊。团建出游。我们相处得更好了，因为我们不再为 AI 系统能轻松完成的工作争论。

## 不确定性是真实的

我不会假装每个人都很开心。

当我不再每天和每个人说话时，一些团队成员感到不确定。CTO 不和我说话意味着什么？我在这个新世界里的价值是什么？这些担忧是合理的。

有些人花在辩论 AI 能否取代他们工作上的时间，比做工作的时间还多。转型期会制造焦虑。我没有一个干净的答案。

我确实有一个原则：我们不会因为工程师引入了一个生产 bug 就开除他。我们改进审查流程。加强测试。增加护栏。对 AI 也一样。如果 AI 犯错，我们构建更好的验证、更清晰的约束、更强的可观测性。

## AI 原生不止于工程

我看到其他公司采用了 AI 优先工程，但其他一切还是手动的。

如果工程几小时就能发布功能，但市场要一周才能公布，市场就是瓶颈。如果产品团队还在跑月度规划周期，规划就是瓶颈。

在 CREAO，我们把 AI 原生运营推到了每个职能：

- **产品发布说明：** AI 从变更日志和功能描述自动生成。
- **功能介绍视频：** AI 生成的动态图。
- **社交媒体每日帖子：** AI 编排并自动发布。
- **健康报告和分析摘要：** AI 从 CloudWatch 和生产数据库生成。

工程、产品、营销、增长在一个 AI 原生工作流中运行。如果一个职能以 Agent 速度运作，另一个以人类速度运作，人类速度的职能会制约一切。

## 给工程师的建议

你的价值正从代码产出转向决策质量。快速写代码的能力每个月都在贬值。评估、批评和引导 AI 的能力在升值。

产品感或品味很重要。你能看一眼生成的 UI 就知道它不对吗？你能看一个架构方案就发现 Agent 漏掉的那个失败模式吗？

我告诉我们 19 岁的实习生：训练批判性思维。学会评估论证、找到漏洞、质疑假设。学会什么是好设计。这些技能会复利增长。

## 给 CTO 和创始人的建议

如果你的产品管理流程比构建时间还长，从那里开始。

在扩展 Agent 之前先构建测试体系。快速的 AI 没有快速验证，就是快速积累的技术债。

从一个架构师开始。一个人构建系统并证明它可行。系统跑起来后再把其他人以操作者角色引入。

把 AI 原生推到每个职能。

预期会有阻力。有些人会反弹。

## 给行业的建议

OpenAI、Anthropic 和多个独立团队收敛到了相同的原则：结构化上下文、专业化 Agent、持久记忆和执行循环。驾驭工程正在成为标准。

模型能力是驱动这一切的时钟。我把 CREAO 的整个转变归因于过去两个月。Opus 4.5 做不到 Opus 4.6 能做的事。下一代模型会进一步加速。

我相信一人公司会变得普遍。如果一个架构师带着 Agent 能做 100 个人的工作，很多公司不需要第二个员工。

大多数我交谈的创始人和工程师还在以传统方式运作。有些在考虑转型。很少有人做到了。

一位记者朋友告诉我，她就这个话题和大约五个人聊过。她说我们比任何人都走得更远："我不认为有人像你们这样彻底重建了整个工作流。"

任何团队都可以用的工具都存在。我们技术栈里没有任何东西是专有的。

**竞争优势是围绕这些工具重新设计一切的决策，以及承担成本的意愿。** 成本是真实的：员工的不确定感、CTO 每天工作 18 小时、高级工程师质疑自己的价值、一个两周的过渡期——旧系统已经拆掉，新系统还没被证明。

我们承担了这个成本。两个月后，数字说话。

我们构建 Agent 平台。我们用 Agent 构建了它。
